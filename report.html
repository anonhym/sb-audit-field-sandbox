Adding @Version to a live MongoDB collection — without a back-fill

How do you introduce Spring Data optimistic locking on documents that already exist with no version field, when a bulk back-fill is forbidden by a business rule?

The problem

Naively adding @Version breaks existing data: a document with no version field loads as version = null, Spring Data decides the entity "is new", routes save() to an insert, and the existing _id throws a duplicate-key error. The usual fix — a one-shot back-fill that sets version = 0 everywhere — is off the table. Each approach below copes with version-less documents at read/write time instead. Tested live on main + six branches.

recommended
If ids are server-assigned (Mongo ObjectId), custom-isnew is the smallest change that works: implement Persistable so the new-vs-existing decision is keyed on the id, leaving the loaded version null; legacy docs migrate to version 0 on first save. If code ever sets the id before saving (UUIDs, business keys), that approach mis-fires — use upsert-all-cases, the only branch that covers every save shape. See client-set ids.

Why it's here

The standard, simplest approach — and exactly the one the business rule rules out. Kept as a reference point so the lazy / per-document strategies can be compared against it.

Why one works and another fails

Every outcome above comes down to one MongoDB equality-match rule used by Spring Data's version-checked update.

The optimistic-update filter decides everything

When Spring Data updates a versioned entity it issues updateFirst({_id, version: <loaded>}, …). Against a legacy document that has no version field:

stored legacy doc
{ _id, name, price,
  stock }
// no version field
{ version: null } ✓ matches
{ version: 0 } ✗ no match

In MongoDB, version: null also matches a document where the field is absent; version: 0 matches only a stored 0.

→ custom-isnew changes only the new/existing decision and leaves the loaded version null → filter matches → migrates.
→ lazy-on-read sets it to 0 → filter misses → fails. No non-null default can win, because only null matches an absent field and null routes back to an insert.
→ explicit-upsert sidesteps the filter entirely by keying on _id.

Edge case: client-assigned ids

A document with id set but version == null is ambiguous: it could be a pre-existing legacy doc (must UPDATE) or a brand-new doc whose id was set in code before saving (must INSERT). You can't tell from the entity alone — and it splits the field in two.

The fix — one idempotent upsert

In upsert-all-cases, the version-null "first write" keys on _id with upsert = true (the only change from explicit-upsert):

boolean firstWrite = (loadedVersion == null);
replaceOne(eq("_id", id), doc, new ReplaceOptions().upsert(firstWrite));
if (!firstWrite && matchedCount == 0) throw new OptimisticLockingFailureException(...);

→ legacy doc present → replaced (migrated). → not present → inserted (new client-id doc). Either way: no duplicate key, no spurious lock failure. Versioned writes keep their version filter, so optimistic locking is intact from the second write onward.

Which write paths maintain @Version?

A separate question: once @Version exists, do all the MongoTemplate write methods keep it correct? Measured live on Spring Data MongoDB 5.1 — and the answer overturns a common assumption: the template update methods already auto-increment the version. Only two paths don't.

note
This was the original "intercept every update and add the increment via AOP" request. It turned out Spring Data already does it for the template update methods (the premise held for older Spring Data, not 5.1). So a blanket aspect would be redundant — the real exposure is just the two gaps below.

The version guard feature/version-guard

Rather than a redundant increment aspect, a small AOP aspect (VersionGuardAspect) warns when the two unguarded paths are used on a versioned entity. Policy: warn & allow. Needs aspectjweaver directly — spring-boot-starter-aop was removed in Boot 4.

bulkOps

Does not auto-increment the version. Add .inc("version", 1) to each bulk update yourself.

[version-guard] bulkOps on Product does NOT auto-increment @Version … Add .inc("version", 1) to each bulk update.

findAndReplace

De-versions the document (drops the field) unless the replacement carries the version. Prefer save() / findAndModify.

[version-guard] findAndReplace on Product does NOT maintain @Version and will DROP the 'version' field …
